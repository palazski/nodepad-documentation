NodePad is built to be self-hosted. On the Enterprise plan, you deploy NodePad into your own infrastructure — Kubernetes via Helm, or a single VM via docker compose — and connect it to your own database, cache, and object storage. NodePad's application layer runs entirely inside your perimeter.

## Two install paths

<Tabs>
  <Tab title="Kubernetes (Helm)">
    For teams running workloads on Kubernetes. Install via the versioned Helm chart published to GitHub Container Registry:

    - Single `image.registry` / `image.repository` value retargets every component to your internal mirror (Nexus, Harbor, Artifactory) without chart edits.
    - Pre-install / pre-upgrade hook Job runs database migrations before any api pods roll. Failures abort the upgrade; existing pods keep serving.
    - Optional Bitnami Postgres / Redis / MinIO sub-charts for self-contained POC installs; production deployments should BYO managed services.

    See the [install guide](/enterprise/operations/install) for the full walkthrough.
  </Tab>
  <Tab title="Single VM (docker compose)">
    For smaller deployments or pilots. An image-only docker compose file ships in the release bundle and pulls pinned immutable tags:

    - One `.env` file captures secrets and configuration.
    - Version pin via a single `VERSION=X.Y.Z` line; `docker compose pull && up -d` to upgrade.
    - Bundled Postgres, Redis, and MinIO for the quick-start path; swap any to a managed service by removing the service and pointing env vars externally.
  </Tab>
</Tabs>

## What you bring

In both postures, you are responsible for:

<Columns cols={2}>
  <Card title="Database" icon="database">
    PostgreSQL 15+. Schema migrations run automatically during install and upgrade.
  </Card>
  <Card title="Cache and queue" icon="zap">
    Redis 6+. Used for WebSocket streaming and background async tasks.
  </Card>
  <Card title="Object storage" icon="hard-drive">
    Any S3-compatible service — AWS S3, Cloudflare R2, DigitalOcean Spaces, MinIO. Stores user-uploaded assets.
  </Card>
  <Card title="Provider API keys" icon="key-round">
    Anthropic and/or OpenAI API keys. Configured once at the deployment level; all users inherit them.
  </Card>
</Columns>

## What NodePad does not do

<Columns cols={3}>
  <Card title="No phone-home" icon="check" type="check">
    Once deployed, NodePad makes no outbound calls to NodePad-managed infrastructure. No update checks, no license phone-home, no usage pings.
  </Card>
  <Card title="No outbound telemetry by default" icon="check" type="check">
    PostHog product analytics is off unless you explicitly set a `NODEPAD_POSTHOG_KEY` pointing at an instance you control.
  </Card>
  <Card title="No data training" icon="check" type="check">
    NodePad does not train models on your conversations, regardless of deployment posture.
  </Card>
</Columns>

## Release cadence and upgrade model

NodePad Enterprise ships semver-tagged container images and Helm charts. Tags `X.Y.Z` and `<git-sha>` are immutable; pin to them in production. Upgrade is `helm upgrade` (K8s) or `docker compose pull && up -d` (VM); migrations run automatically. Release notes live at `docs.node-pad.com/changelog`. Each release also has a [GitHub Release page](https://github.com/palazski/nodepad-api/releases) with the offline bundle attached.

See [Upgrade & rollback](/enterprise/operations/upgrade) for the full playbook.
