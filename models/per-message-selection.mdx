---
title: "Per-message model selection"
description: "Use a workspace default when it helps, then override the model for any specific send."
---

NodePad does not lock a branch to one model forever. You can keep a sensible default for the workspace and still override the model for a specific send whenever the branch needs something different.

## The basic pattern

<Steps>
  <Step title="Set a workspace default">
    Choose the model you want most new work in this workspace to use.
  </Step>
  <Step title="Override when a branch needs it">
    Before sending a message, pick another model for just that send if the branch now needs a different style of output or reasoning depth.
  </Step>
  <Step title="Keep the branch structure stable">
    If you want a real comparison, branch first and then use a different model in the new branch.
  </Step>
</Steps>

## Good reasons to override the model

<AccordionGroup>
  <Accordion title="You need a second opinion">
    Branch from the same message and route the new branch to another model.
  </Accordion>
  <Accordion title="The branch changed jobs">
    A branch that started as ideation may now need critique, summarization, or deeper reasoning. Override the model for that next send instead of starting over.
  </Accordion>
  <Accordion title="You care about speed in one branch and depth in another">
    Keep one branch on a lighter model for rapid iteration while another branch runs on a heavier model for analysis.
  </Accordion>
  <Accordion title="You want a real A/B comparison">
    Use the same source message in two branches, change only the model, and compare the outputs side by side.
  </Accordion>
</AccordionGroup>

<Note>
  Model overrides are most useful when you change one thing at a time. If the prompt and the branch history change too, you lose the clean comparison.
</Note>
