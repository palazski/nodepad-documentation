---
title: "Nodes"
description: "Nodes are the individual messages and replies that make up a branch in NodePad."
---

A node is one message on the canvas. User messages, assistant replies, and in-progress drafts all behave like nodes in the graph. Most NodePad actions start by selecting a node or branching from one.

## What a node can contain

<AccordionGroup>
  <Accordion title="Message content">
    NodePad renders message text with Markdown, so lists, code blocks, and links stay readable inside the graph.
  </Accordion>
  <Accordion title="Attachments">
    A node can carry uploaded files, images, and tool-call artifacts. Those attachments stay part of the node's context footprint.
  </Accordion>
  <Accordion title="Model metadata">
    Assistant nodes show which model produced the reply. Workspace defaults control the starting point, but individual sends can override them.
  </Accordion>
  <Accordion title="References">
    A node can reference other messages with `@` mentions. The visible token is only the UI layer; NodePad stores the actual cross-message links separately.
  </Accordion>
</AccordionGroup>

## How nodes become branches

Nodes connect in sequence inside a branch. A simple branch looks like:

1. your message
2. the assistant reply
3. your next message
4. the next reply

When you branch from a node, NodePad creates a new path from that point without changing the original branch.

## Actions you can take from a node

<CardGroup cols={3}>
  <Card title="Branch from here" icon="git-fork" href="/concepts/forking">
    Start a new branch from the exact context at that point.
  </Card>
  <Card title="Edit or re-run" icon="pencil">
    Depending on the node type and position, you can edit, re-send, or re-run from that point.
  </Card>
  <Card title="Reference or merge" icon="at-sign" href="/concepts/references">
    Pull this node into another message through an `@` reference or a merge flow.
  </Card>
</CardGroup>

## Important behaviors to know

<AccordionGroup>
  <Accordion title="Branches preserve history">
    Branching does not rewrite the original path. It creates a new one.
  </Accordion>
  <Accordion title="References are explicit">
    Mentioning `@B1-2` in text is not enough on its own. The reference picker records the actual linked node so NodePad knows what to include.
  </Accordion>
  <Accordion title="Node IDs are useful">
    You can copy a node ID from the context menu. That is handy when you want to reference a specific message cleanly.
  </Accordion>
  <Accordion title="Text notes are different">
    Text notes are not conversation nodes. They are standalone annotations on the canvas.
  </Accordion>
</AccordionGroup>

<Tip>
  If you are unsure whether a follow-up belongs in the current branch, branch first. It is cheaper to keep two clean paths than to untangle one noisy one later.
</Tip>
