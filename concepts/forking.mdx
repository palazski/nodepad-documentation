---
title: "Forking"
description: "Branch from any message so you can explore a new path without overwriting the current one."
---

Forking is how NodePad turns one conversation into two. Instead of asking every follow-up in the same branch, you can split at a specific message and test another direction beside it.

## When to fork

<CardGroup cols={2}>
  <Card title="Get a second opinion" icon="messages-square">
    Ask another model the same question before you commit to a decision.
  </Card>
  <Card title="Chase a tangent" icon="route">
    Explore a side question without polluting the branch you still care about.
  </Card>
  <Card title="Compare prompts" icon="split">
    Send two different follow-ups from the same point and see which framing works better.
  </Card>
  <Card title="Preserve a checkpoint" icon="shield">
    Split before a risky change so the original path stays intact.
  </Card>
</CardGroup>

## How branching works

<Steps>
  <Step title="Pick the message to branch from">
    Right-click the node that represents the context you want to preserve.
  </Step>
  <Step title="Choose Branch from here">
    NodePad creates a new draft connected to that source node.
  </Step>
  <Step title="Send a new follow-up">
    The new draft starts from the same history up to that point, then diverges based on what you send next.
  </Step>
  <Step title="Compare both branches on the canvas">
    Because both branches stay visible, you can compare results without switching tabs or losing provenance.
  </Step>
</Steps>

## Branching patterns that work well

<AccordionGroup>
  <Accordion title="Same question, different model">
    Branch from a user message, keep the wording the same, and route the branch to another model. This is useful when you want a true second read on the same prompt.
  </Accordion>
  <Accordion title="Same branch, tighter scope">
    Branch from a broad response and ask one narrower question in the new branch. The main branch stays high level while the new one goes deep.
  </Accordion>
  <Accordion title="Re-send versus branch">
    Re-send is useful when you want to try the same message again. Branching is better when you want an alternate path you plan to keep around.
  </Accordion>
  <Accordion title="Branch before merge work">
    If you know a branch is going to become source material for another message later, branching early keeps those sources cleaner and easier to reference.
  </Accordion>
</AccordionGroup>

<Info>
  Branches are explicit graph structure. They are not just copied text in a new chat window.
</Info>
